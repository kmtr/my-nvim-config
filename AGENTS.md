# Repository Guidelines

## Project Structure & Module Organization

This repository is a personal Neovim configuration. `init.lua` is the entrypoint and loads numbered Lua modules from `lua/` in a fixed order. Keep global options in `lua/001-opts.lua`, keymaps in `lua/002-keys.lua`, plugin declarations in `lua/000-plugins.lua`, plugin-wide setup in `lua/1xx-*.lua`, LSP/debugging setup in `lua/2xx-*.lua`, and filetype-specific setup in `lua/3xx-*.lua`. The README documents user-facing setup and key mappings. `lazy-lock.json` is generated by lazy.nvim and ignored by Git.

## Build, Test, and Development Commands

- `ln -s $(pwd) ~/.config/nvim`: install this checkout as the active Neovim config.
- `nvim`: start Neovim and let `lazy.nvim` install missing plugins.
- `:Lazy`: open the lazy.nvim plugin console.
- `:Lazy update`: update installed plugins locally.
- `nvim --headless "+luafile init.lua" +qa`: smoke-load the configuration without opening the UI.
- `luac -p lua/001-opts.lua`: syntax-check a Lua file; repeat for edited modules.

## Coding Style & Naming Conventions

Use Lua for configuration files and four-space indentation, matching the existing option files. Prefer `vim.opt` for options, `vim.keymap.set` for mappings, and `require(...)` for module loading. Name modules with the existing numeric prefix convention, for example `102-flash.lua` or `301-rust.lua`, so load order stays obvious. Keep plugin-specific configuration close to the plugin module instead of growing `init.lua`.

## Testing Guidelines

There is no formal test suite. For each change, run a Lua syntax check on touched files and smoke-load Neovim headlessly. For plugin, LSP, or keymap changes, also test interactively with `nvim` and verify the affected command or mapping. If adding a language server, document any external installation requirement in `README.md`.

## Commit & Pull Request Guidelines

Recent commits use short, imperative, lowercase summaries such as `update plugins` and `move flash shortcut keys`. Follow that style and keep each commit focused on one behavior or configuration area. Pull requests should explain the user-visible change, list manual verification steps, and note any new external dependency such as `ripgrep`, `gopls`, or `rust-analyzer`.

## Security & Configuration Tips

Do not commit machine-local plugin output, lockfiles generated by lazy.nvim, secrets, or paths specific to one workstation. Keep generated directories such as `plugged/`, `plugin/`, and `lazy-lock.json` out of version control.
